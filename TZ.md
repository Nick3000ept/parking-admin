# ТЗ: Веб-приложение учёта работ по паркингу

## 1. Назначение

Веб-приложение для подрядчиков и админа — отметка факта выполнения работ по ~2000 помещений **одного** паркинга. Каждый пользователь заходит по своей ссылке-токену; интерфейс — единый, права различаются.

**Целевая платформа — десктоп** (мобильную адаптацию не делаем). **Один токен = вся команда подрядчика** (не персональный, идентифицируем подрядчика как юр.лицо).

**Две роли в UI:**
- **Подрядчик** — видит и отмечает только свои назначения; чужие назначения видны read-only с именем подрядчика.
- **Админ** (специальная запись в листе `Подрядчики` с `ИД_подрядчик = admin`) — видит и отмечает любые работы за любого подрядчика. Также продолжает вести справочники, реестр и назначения в Google Sheets — UI не покрывает редактирование структуры.

## 2. Технический стек

- **Frontend**: один `index.html` (vanilla JS, без сборки)
- **Хостинг**: GitHub Pages, репозиторий `parking-admin` (github.com/Nick3000ept)
  - URL: `https://nick3000ept.github.io/parking-admin/`
  - Деплой ветки `main`, корень
- **Backend**: Google Apps Script (Web App), эндпоинты `doGet` / `doPost`, ответы JSON
- **БД**: Google Sheets — исходный файл с реестром помещений (`1zB1IQJlla93txI9o-sH-mxD3CeeO0Ugkv8mHPDzcqiI`). Колонки работ и СП добавляются в тот же лист реестра, в конец после исходных столбцов. **Отдельная проектная копия не создаётся.** Резервная копия раз в неделю вручную (см. §10).
- **Хранение кода**: `index.html`, `script.gs`, `TZ.md`, `README.md` — в git-репо

## 3. Структура Google Sheets

### Лист `Главный` — реестр + назначения + факт в одной таблице

Одна строка = одно помещение. Слева реестровые поля, справа парные колонки на каждую работу.

**Реестровые колонки (A–L)** — заполняет/копирует админ при импорте помещений:

| Колонка | Поле | Источник |
|---|---|---|
| A | Корпус | из исходного реестра |
| B | Этаж | из исходного реестра |
| C | Номер (PK, `7.-3.01`) | из исходного реестра |
| D | Наименование | из исходного реестра |
| E | Площадь, м.кв. | из исходного реестра |
| F | Периметр, м.п. | из исходного реестра |
| G | Площадь стен | из исходного реестра |
| H | Категория помещения | из исходного реестра |
| I | Наименование чертежа | из исходного реестра |
| J | Захватка | из исходного реестра |
| K | Комментарий (админский, подрядчик не видит) | вручную |
| L | Распределение для КС (группа для применимости работ) | из исходного реестра |

**Парные колонки на каждую работу (M+), повторяются для всех работ из листа `Работы` в порядке `порядок`:**

- `<Название работы>` — *дата факта*. Хранится в Sheets как **`Date` (формат ячейки = `Дата`, отображение `ДД.ММ.ГГГГ`)**. Пусто = не сделано.
- `<Название работы> СП` — *подрядчик* (имя из листа `Подрядчики`). Пусто = работа не нужна в этом помещении.

Пример для работы «Гидроизоляция полов»:

| ... | Гидроизоляция полов | Гидроизоляция полов СП | Заливка стяжки | Заливка стяжки СП | ... |
|---|---|---|---|---|---|
| ... | 20.05.2026 | Спектр | | Спектр | ... |
| ... | | | 20.05.2026 | Спектр | ... |

**Правила колонок:**
- A–L редактирует только админ в Sheets
- Колонки `*СП` (назначение) — ведёт админ в Sheets
- Колонки дат — заполняются подрядчиком через UI (либо админом в Sheets вручную)
- **Применимость работы** к помещению = «есть значение в колонке СП». Нет СП = работа не нужна тут.
- **Инвариант**: дата не может быть заполнена при пустом СП. Условное форматирование подсвечивает нарушение красным.

**Правила работы GAS с листом `Главный` (критично, реестр — боевые данные):**
- **GAS никогда не пишет в столбцы A–L** (реестровая часть). Только M+ (колонки работ и СП).
- **GAS не добавляет и не удаляет строки** в листе. Только обновляет существующие ячейки в M+.
- В коде это enforce'ится явной проверкой: `if (colIndex <= 12) throw new Error('Запись в реестр запрещена')`. Любая ошибка вычисления индекса = падение с понятной ошибкой, а не порча данных.
- Лист с реестром в файле называется ровно **`Главный`** (точное имя для `getSheetByName`).
- На момент старта разработки последняя занятая колонка работ — `Заливка топпинга СП` (после неё новых работ нет, можно добавлять справа без конфликтов).

### Лист `Работы` — справочник, ведётся вручную

| A: Полное название | B: Поверхность | C: Название для площадки | D: ИД_работы |
|---|---|---|---|
| Гидроизоляция полов | Полы | Гидроизоляция | down_hydro |
| Заливка стяжки | Полы | Стяжка | down_st |
| Укладка плитки | Полы | Плитка | down_kerama |
| Покраска потолка | Потолки | Покраска потолок | up_color |
| Штукатурка стен | Стены | Штукатурка | wall_stukat |
| Покраска стен | Стены | Покраска стены | wall_paint |
| Монтаж двери | Двери | Двери | doors |
| Заливка топпинга | Полы | Топпинг | down_topping |

- **A: Полное название** — официальное имя работы. **Должно точно совпадать с заголовком колонки в `Главный`** (например `Гидроизоляция полов` / `Гидроизоляция полов СП`). При переименовании работы — переименовать одновременно тут и в `Главный`.
- **B: Поверхность** — группа работ (`Полы` / `Стены` / `Потолки` / `Двери`). Используется для группировки колонок в UI подрядчика с подзаголовками.
- **C: Название для площадки** — короткое имя для заголовков колонок UI подрядчика (узкие столбцы на мобилке). Можно править без последствий для системы.
- **D: ИД_работы** — стабильный латинский идентификатор для API между фронтом и GAS. **Неизменяемый.** Используется в URL/JSON запросах.
- Порядок строк в листе = порядок колонок в `Главный` и в UI (нет отдельной колонки «порядок»).

### Лист `Подрядчики` — справочник, ведётся вручную

| A: Подрядчик | B: ИД_подрядчик | C: Статус | D: Токен |
|---|---|---|---|
| Овчинников | ovchi | Активен | 81bc3299c7ceb6fb |
| Спектр | spectr | Активен | ce3d0facb5187091 |
| Альфа Строй | alfa | Активен | 8ca318c5521b5b66 |
| Админ | admin | Активен | <admin_token_16hex> |

- **A: Подрядчик** — имя как пишется в колонках `*СП` листа `Главный`. **Primary key** для связи с реестром. Может содержать пробелы и кириллицу.
- **B: ИД_подрядчик** — стабильный латинский ID для API/кода. Используется в логах и при сравнении на бэке. Регистр единообразный (строчные). **Зарезервированное значение: `admin` = роль администратора** (бэк даёт права на отметку любых работ за любого подрядчика).
- **C: Статус** — текстовое значение. **`Активен` = доступ открыт; любое другое значение (или пусто) = доступ закрыт** (можно использовать `Уволен`, `На паузе` и т.п. — бэк сравнивает строго на равенство `"Активен"`).
- **D: Токен** — 16 hex-символов, генерируется формулой (см. §7). Уникальный ключ доступа подрядчика, передаётся в URL.
- Колонка «какие работы делает» **не нужна** — применимость определяется фактическими записями в `Главный.*СП`.
- Запись с `ИД_подрядчик = admin` — не должна попадать в колонки `*СП` листа `Главный` (админ ничего не делает, он только проверяет/корректирует).

### Лист `Типы помещений` — справочник типов с цветами и группировкой

Используется для (1) цветовой индикации карточек в матрице (см. §5), (2) группирующего фильтра «Тип помещения» в шапке UI.

| A: Тип_помещения_детально | B: Тип_помещения_панель | C: Цвет |
|---|---|---|
| Кладовая | Кладовая | Желтый |
| Тамбур-шлюз (ЛХ/ПБЗ) | ТШ | (пусто) |
| Лестничная клетка | ЛК | (пусто) |
| Архив | Прочие помещения | (пусто) |
| Душевая охраны | Прочие помещения | (пусто) |
| Помещение КНС | Тех помещения | Серый |
| Автостоянка | Автостоянка | (пусто) |
| Лифтовой холл | ЛХ | (пусто) |
| Тамбур | Тамбур | (пусто) |

- **A: Тип_помещения_детально** — точное побайтное совпадение со значениями в `Главный!D` (`Наименование`). Каждое уникальное наименование из реестра должно иметь строку здесь.
- **B: Тип_помещения_панель** — укрупнённая категория для фильтра в UI. Несколько детальных типов могут мапиться в одну панельную категорию (например, все «Прочие помещения» агрегируются под одной чипсой фильтра).
- **C: Цвет** — текстовое название цвета. Фронт мапит на hex через статический пресет в коде. Пустое значение или название вне пресета — дефолтный светло-серый (`#E5E7EB`).
  - **Зелёный цвет (любой оттенок) НЕЛЬЗЯ использовать** для типа помещения — он зарезервирован за индикацией «работа готова» (см. §5). Если в Sheets будет введено `Зелёный` — фронт игнорирует и применяет дефолтный серый.
  - Пресет цветов в коде (расширяемый): Жёлтый, Серый, Оранжевый, Красный, Синий, Розовый, Фиолетовый, Голубой, Коричневый. Конкретные hex подбираются при разработке так чтобы белый текст номера читался поверх.
- **Растущий лист** — появилось новое уникальное `Наименование` в реестре → админ добавляет строку. Если строки нет — фронт показывает карточку с дефолтным серым и логирует warning в консоли GAS.

**Связь типов с реестром:** GAS при формировании JSON делает lookup на лету: для каждой строки `Главный` по полю `Наименование` ищет соответствующую строку в `Типы помещений`, добавляет в JSON помещения два поля: `тип_панель` и `цвет_hex`. Колонок в `Главный` не добавляем — `Главный` остаётся чистым.

## 4. Авторизация и роли

Без паролей. Первичный доступ по ссылке: `https://nick3000ept.github.io/parking-admin/?t=<token>`

- **Подрядчик**: токен → запись в `Подрядчики` (если `Статус == "Активен"`) → его `Подрядчик` (имя). Видит **все помещения** в матрице (для пространственной ориентации), но **активно работать может только с теми, где он назначен** (`Главный.<работа> СП[Номер] == <его имя>`). Чужие назначения видны как read-only с именем подрядчика (в попапе).
- **Админ**: токен с `ИД_подрядчик = admin` (если `Статус == "Активен"`). Видит все помещения, **может ставить/снимать отметки за любого подрядчика** в любой работе где есть назначение. Назначать СП — по-прежнему только в Sheets.
- Без токена / невалидный / `Статус != "Активен"` → экран «Доступ закрыт».

**Жизненный цикл токена в браузере:**
- При первом заходе по `?t=<token>` фронт сохраняет токен в `localStorage` под ключом `parking_token` и **удаляет `?t=` из URL** (через `history.replaceState`). Защита от утечки через скриншоты, history и расшаренные ссылки.
- При последующих заходах токен берётся из `localStorage`. URL остаётся чистым.
- Кнопка «Выйти» в шапке UI — очищает `localStorage` и редиректит на экран «Доступ закрыт» с пояснением «откройте ссылку, выданную администратором».

**Ротация токена при утечке** (см. §7.7 и §7.14):
- Админ в Sheets перегенерирует значение в `Подрядчики.Токен` через формулу
- Старый токен мгновенно перестаёт работать (бэк не находит совпадения)
- Новую ссылку отправляют легитимному подрядчику
- При следующем заходе со старым токеном — экран «Доступ закрыт»

**Безопасность** — проверка на бэке. На каждый запрос GAS сверяет:
1. Токен валиден и `Статус == "Активен"`
2. Для write-запроса (Номер, ИД_работы):
   - Если **подрядчик**: в листе `Главный` строка с этим Номером, колонка `<название_работы> СП` содержит **ровно его имя**. Иначе — отказ.
   - Если **админ** (`ИД_подрядчик == 'admin'`): достаточно чтобы колонка `<название_работы> СП` была непустой (есть кому назначено). Иначе — отказ (не назначаем работы через UI, только в Sheets).
3. **Менять СП запрещено всем** (включая админа в UI) — только через Sheets. UI всегда работает только с колонкой даты.
4. **Редактировать дату нельзя** — только полный clear+set (бинарно «есть/нет»).

#### Шапка
- Название проекта (`Паркинг СБ3`)
- Прогресс **в зависимости от выбранной работы**:
  - Работа выбрана: «Сделано X из Y по работе `<название>` (мои назначения)»
  - Работа не выбрана: «Сделано X из Y по всем моим работам»
  - Для админа — «по всем работам всех подрядчиков»
- Кнопка «Обновить» — форсированный `doGet`
- Кнопка «Выйти» — очищает localStorage, редирект на «Доступ закрыт»

#### Фильтры
- **Захватка** — чипсы из уникальных значений колонки `Главный!J`. Multi-select.
- **Корпус** — чипсы из уникальных значений колонки `Главный!A`. Сортировка: сначала числа по возрастанию (7, 8, …, 12), потом буквенные (П1, П2). Multi-select.
- **Этаж** — чипсы из уникальных значений колонки `Главный!B`. Сортировка по возрастанию (-3, -2, -1). Multi-select.
- **Тип помещения** — чипсы из уникальных значений колонки `Тип_помещения_панель` листа `Типы помещений`. Multi-select. Сортировка алфавитом (или по частоте использования — на усмотрение разработки).
- **Работа** — dropdown из листа `Работы` (для подрядчика — только работы где он назначен хотя бы где-то; для админа — все). Single-select. Имеет пустое значение «(все)» = режим без выбранной работы.
- **Поиск по номеру** — текстовое поле, частичное совпадение по `Главный!C`.
- **Поиск по назначению** — текстовое поле, частичное совпадение по `Главный!D` («Наименование») или `Главный!L` («Распределение для КС»).
- **Тогл «Скрыть готовые»**:
  - Работа выбрана: скрывает помещения где дата выбранной работы заполнена
  - Работа не выбрана: тогл disabled (нет смысла «готовый по чему»)
- **Тогл «Скрыть чужие»**:
  - Для подрядчика: скрывает помещения где он не назначен на выбранную работу (или ни на одну, если работа не выбрана)
  - Для админа: скрывает помещения где не назначен никто на выбранную работу (или совсем без назначений)
- **Кнопка «Очистить»** — сбрасывает все фильтры в дефолт.

#### Матрица этаж × корпус

- **Layout**: CSS Grid. Строки — этажи (`-1`, `-2`, `-3` сверху вниз). Колонки — корпуса (отсортированные).
- **Контейнер матрицы**: занимает весь экран минус шапка и фильтры (`height: calc(100vh - <header_height>)`).
- **Размер ячеек — адаптивный** через `grid-template-rows: repeat(N, 1fr)` и `grid-template-columns: repeat(M, 1fr)`, где N — число видимых этажей после фильтрации, M — число видимых корпусов. Изменение фильтра меняет N/M → ячейки автоматически перерасчитываются. Никакого JS на ресайз — чистый CSS.
  - 3 этажа × 7 корпусов = 21 маленькая ячейка
  - 1 этаж × 7 корпусов = 7 широких колонок на всю высоту
  - 3 этажа × 1 корпус = 3 строки на всю ширину
  - 1 этаж × 1 корпус = одна большая ячейка на весь экран
- **Заголовки**:
  - Над колонками — номер корпуса
  - Слева у строк — номер этажа
- **Ячейка матрицы (этаж × корпус)**:
  - Адаптивный размер (см. выше)
  - Внутри ячейки — flex wrap карточек помещений (направление left-to-right, top-to-bottom)
  - `overflow-y: auto` — скролл внутри ячейки если карточек больше чем влезает
  - Размер самих карточек остаётся фиксированным (~80×40px) — не растягиваются под размер ячейки
- **Карточка помещения**:
  - **Все карточки одинакового размера** ~80×40px (не растягиваются, не сжимаются)
  - Содержимое — только номер помещения (`7.-3.01`)
  - Базовый фон без выбранной работы — цвет из листа `Типы помещений` (lookup по `Наименование` → `Цвет`). Если тип не найден / цвет пуст — дефолтный светло-серый.
  - **Состояния зависят от режима (работа выбрана / не выбрана):**

  **Режим «работа НЕ выбрана»** — карточки информативные, чекбоксов нет, цвет = тип помещения:

  | Состояние | Цвет фона | Интерактив |
  |---|---|---|
  | Любое помещение | Цвет типа из `Типы` | Только клик по номеру → попап |

  **Режим «работа выбрана»** — каждая карточка отражает статус выбранной работы:

  | Состояние | Цвет фона | Чекбокс |
  |---|---|---|
  | Назначено мне, не готова | Цвет типа | Пустой, кликабельный → ставит дату |
  | Назначено мне, готова | **Зелёный (яркий)** | ✓ кликабельный → confirm → снимает |
  | Назначено чужому, не готова | Светло-серый | Нет (read-only) |
  | Назначено чужому, готова | **Зелёный приглушённый** (opacity 0.5) | Нет (read-only) |
  | Не назначено никому | Светло-серый | Нет |
  | Для админа: любое назначение | Цвет типа / зелёный | Чекбокс активен везде где есть СП |

  - Клик по самому номеру (не по чекбоксу) → **попап с деталями** (см. ниже). Работает в любом состоянии.
  - В попапе строки с готовыми работами также подсвечены зелёным фоном для консистентности.

#### Попап помещения (детальный режим)

Привязан к карточке (всплывает рядом). Закрывается кликом вне, Esc, или крестиком.

```
┌─────────────────────────────────────────┐
│ 7.-3.01                              ✕  │
│ Кладовая                                │
│ Распределение для КС: Кладовые          │
│ Площадь: 4,19 м²   Периметр: 8,79 м     │
├─────────────────────────────────────────┤
│ Работы:                                 │
│  ☑ Гидроизоляция полов  • Спектр        │
│    20.05.2026                           │
│  ☐ Заливка стяжки       • Спектр        │
│  ☑ Укладка плитки       • Овчинников    │
│    15.05.2026                           │
└─────────────────────────────────────────┘
```

**Что показывается:**
- Заголовок: номер + наименование + Распределение для КС
- Площадь, периметр
- Список работ **где есть назначение** (`*СП` непусто). Работы без назначения не показываем.
- Для каждой работы:
  - Чекбокс (отмечен если дата стоит)
  - Название работы (`Полное название` из листа `Работы`)
  - Имя назначенного подрядчика
  - Дата под чекбоксом если отмечено

**Интерактив:**
- **Подрядчик**: чекбоксы активны только на своих работах. Чужие — read-only с серым фоном.
- **Админ**: все чекбоксы активны.
- Клик по пустому чекбоксу → ставит дату (сегодня сервера), без confirm
- Клик по отмеченному → confirm «Снять отметку?» → снимает дату

#### Пограничные состояния UI
- **Подрядчику не назначено ни одной ячейки**: пустой экран с сообщением «Пока нет назначенных работ» и кнопкой «Обновить». Матрица не рисуется.
- **Запрос на установку даты упал (сеть)**: попап «Не удалось сохранить, повторить?» с кнопками Повторить / Отмена. Локально не пишем (избегаем рассинхрона).
- **Токен невалиден / Статус ≠ Активен**: экран «Доступ закрыт» с просьбой обратиться к администратору
- **`Распределение для КС` пусто** у помещения: карточка с дефолтным серым цветом, в фильтре поиска по назначению такие помещения помечены как `(не указано)`
- **Тип помещения не найден в листе `Типы`**: дефолтный серый, в консоли GAS warning
- **Загрузка идёт**: над матрицей — скелетон или спиннер «Загружаем данные…»

#### Чего нет в UI
- Управления реестром / справочниками (в Sheets)
- Назначения подрядчиков (колонки `*СП`) — даже у админа в UI
- Колонки `K: Комментарий` (админская, в Sheets)
- Комментариев подрядчика к отметкам (только дата)
- Редактирования даты — только бинарно «есть/нет», дата = сегодня сервера
- Мобильной/планшетной версии

## 6. Производительность и параллелизм

### Объём данных
- ~2000 помещений × ~10 работ × парные колонки (дата + СП) = до 40k значений по работам + ~12 колонок реестра
- Сырой JSON для матрицы — оценочно ~1–1.5 MB. С gzip от GAS — ~300–500 KB по сети. Терпимо.

### Бэк (GAS)
- **doGet**: читает `Главный` целиком (один `getValues`), листы `Работы`, `Подрядчики`, `Типы помещений`. Делает lookup типа помещения по `Наименование` → `Тип_помещения_панель` и `Цвет`. Сериализует в JSON. На фронт отдаёт **всё** (для построения матрицы и попапов), без фильтрации по пользователю — права проверяются на write, не на read. Ожидаемое время — 3–8 сек, отображается спиннер.
  - Альтернатива «отдавать только колонки нужные пользователю» **рассматривалась и отклонена** — она усложняет логику и не даёт большого выигрыша, потому что в попапе всё равно нужны чужие назначения.
- **doPost**: точечное обновление одной ячейки даты. Цель — <1 сек.
- **LockService** в GAS: каждая write-операция оборачивается в `LockService.getScriptLock().waitLock(10000)`. Гарантирует последовательную запись если два сотрудника одной команды кликают одновременно.
- **Идемпотентность** (см. §7.5): повторная установка даты на уже отмеченную ячейку — no-op. Повторное снятие на уже пустую — no-op.
- Без поллинга/автообновления — только по кнопке «Обновить».

### Фронт
- **2000 карточек в матрице** — на десктопе с современным браузером CSS Grid с фиксированным размером карточек справляется без виртуализации. Не закладываем виртуализацию сразу. **Если на тестах с реальным объёмом будут лаги — добавим IntersectionObserver на ячейки матрицы** (рендер карточек только в видимых ячейках).
- **Фильтрация** — пересчёт `display: none / block` для карточек. При больших фильтрах (например выбор одного корпуса) — изменение `grid-template-columns` контейнера матрицы. Не должно тормозить, но проверим на тестах.
- **Попап** — отрисовывается по требованию, в DOM один экземпляр.

### Кеш в `localStorage`
- Ключ: `parking_data_<token>`
- Содержимое: полный JSON ответа `doGet` + `version_hash` (хеш от заголовков `Главный` + содержимого `Работы`/`Подрядчики`/`Типы`)
- Стратегия: при открытии страницы UI рендерит из кеша **мгновенно**, параллельно делает `doGet`. Если `version_hash` совпал — фоновое обновление дат; если не совпал — сброс кеша и полный ререндер (изменилась структура колонок или цветовая палитра)
- Кнопка «Обновить» — форсированный `doGet` с показом спиннера.

## 7. Правила разработки

1. **Источник правды о работах** — лист `Работы`, колонка `ИД_работы` = стабильный латинский ID для API.
2. **Заголовки колонок работ в `Главный`** должны точно совпадать с `Работы.Полное название` (например `Гидроизоляция полов` / `Гидроизоляция полов СП`). При переименовании работы — обновить оба места одновременно. Колонки `Поверхность` и `Название для площадки` можно править без последствий.
3. **Источник правды о подрядчиках** — лист `Подрядчики`, `имя` = primary key. Имена в `Главный.*СП` должны совпадать побайтно.
4. **Валидация прав на бэке**: см. §4. Клиенту не доверяем.
5. **Идемпотентность**: повторная отправка той же даты на ту же ячейку не дублирует.
6. **Оффлайн**: при обрыве сети — попап «не удалось сохранить, повторить?». Локально не пишем.
7. **Генерация токенов**: формула в листе `Подрядчики`, колонка `Токен` (16 hex-символов):
   ```
   =LOWER(DEC2HEX(RANDBETWEEN(1,4294967295),8))&LOWER(DEC2HEX(RANDBETWEEN(1,4294967295),8))
   ```
   После генерации — Ctrl+Shift+V → значения, чтобы не пересчиталось.
8. **Деплой**: `git add . && git commit && git push` → GitHub Pages обновится через ~1 мин. GAS деплоится отдельно через `Deploy → Manage deployments`.
9. **Data Validation в `Главный`**: колонки `*СП` — выпадающий список из `Подрядчики.имя`. Защита от опечаток.
10. **Условное форматирование в `Главный`**:
    - Серый фон колонки `<работа>` когда `<работа> СП` пуста — визуально «работа не нужна тут»
    - Красный фон ячейки даты если СП пуст (нарушение инварианта)
11. **При импорте новых помещений** из исходного реестра — копируются только колонки A–J и L. Колонки K (комментарий) и работ (M+) остаются пустыми, заполняются в `Главный`.
12. **Добавление новой работы в систему** (рутинная операция, список работ растущий):
    1. Добавить строку в листе `Работы`: `Полное название`, `Поверхность`, `Название для площадки`, латинский `ИД_работы`
    2. В листе `Главный` добавить **две новые колонки в конец**: `<Полное название>` и `<Полное название> СП`. Порядок колонок в `Главный` должен соответствовать порядку строк в `Работы`.
    3. Назначить подрядчиков в колонке `*СП` где работа применима
    4. UI подрядчика подхватит новую работу автоматически при следующем «Обновить»
    
    Удалять колонки уже использованных работ нельзя (потеряются исторические даты). Работу можно «деактивировать» удалением строки в `Работы` — в `Главный` колонки останутся, UI их игнорирует.
13. **Добавление нового подрядчика** (рутинная операция, список растущий):
    1. Добавить строку в листе `Подрядчики`: `Подрядчик` (имя), `ИД_подрядчик` (латиница строчными), `Статус = Активен`, `Токен` (через формулу, потом Ctrl+Shift+V)
    2. Имя автоматически появится в выпадающих списках колонок `*СП` листа `Главный` (Data Validation на диапазоне `Подрядчики!A:A`)
    3. Передать подрядчику ссылку с его токеном
    
    «Уволить» подрядчика — поставить `Статус = Уволен` (или любое значение, отличное от `Активен`). Назначения в `Главный` остаются (исторические данные).
14. **Ротация токена** (при подозрении на утечку):
    1. В листе `Подрядчики`, ячейка `Токен` нужного подрядчика — вставить формулу из §7.7 заново
    2. Ctrl+Shift+V → значения (превратить в строку)
    3. Скопировать новый токен, отправить подрядчику новую ссылку
    4. Старая ссылка перестаёт работать мгновенно — старый кеш в браузере у злоумышленника даст одну попытку чтения, после чего бэк отклонит токен. **Локальные правки злоумышленник сделать не сможет** (бэк всегда сверяет токен на каждый запрос).
15. **Смена СП в Главный при существующей дате**: дата НЕ очищается. Это считается передачей записи новому подрядчику. UI старого подрядчика перестаёт видеть эту ячейку (она больше не его), новый подрядчик видит её уже отмеченной и может снять/перепоставить.
16. **Дата ставится сервером**: при `doPost` действия `setDate` GAS подставляет `new Date()` в таймзоне `Europe/Moscow`. Клиент не передаёт дату.
17. **Удаление помещения из реестра**: если админ удаляет строку в листе `Главный`, все проставленные даты и назначения по этому помещению теряются вместе со строкой. Бэкап — ответственность админа (см. §11).

## 8. Чего НЕ делаем (выкинуто намеренно)

- Промежуточные статусы — только «есть дата»/«нет даты». **Подтверждение/приёмка админом не предусмотрены** (дата подрядчика = факт)
- **Редактирование даты подрядчиком** — только бинарно поставить/снять, всегда сегодняшняя
- Фото-доказательства
- Лог действий (кто когда что нажал) — не ведём. При спорах источник — Sheets, версии Google можно посмотреть через `Файл → История версий`
- Уведомления (push, email, Telegram)
- **Мобильная и планшетная версии** — только десктоп (min-width 1024px)
- PWA / offline-режим
- Регистрация / сброс доступа подрядчиками (только админ ротирует токен)
- Управление справочниками из UI (только в Sheets)
- **Назначение подрядчиков (колонки `*СП`) через UI** — даже у админа. Только в Sheets.
- Отдельная матрица применимости работ (применимость = есть подрядчик в `*СП`)
- Отдельные нормализованные листы `Назначения` / `Статусы` (всё в `Главный`)
- Комментарии подрядчика к отдельной отметке (общий комментарий ведёт админ в колонке K)
- Колонка «какие работы делает подрядчик» в листе `Подрядчики`
- **Мультитенантность** — система только под один паркинг
- **Персональные токены сотрудников** — токен на команду подрядчика, не на физлицо

## 10. Риски и операционные меры

| Риск | Меры |
|---|---|
| Утечка токена подрядчика (скриншот/расшаренная ссылка) | Очистка `?t=` из URL после первого входа (§4). Ротация через перегенерацию формулы (§7.14). Влияние ограничено — злоумышленник может менять даты только на ячейках одного подрядчика |
| **Утечка админского токена** — высокий риск, злоумышленник может менять любые даты | Админ-токен хранится только у заказчика, не передаётся через мессенджеры/email. При подозрении на утечку — немедленная ротация. Не использовать admin-токен на чужих устройствах |
| Случайное удаление строки в `Главный` | Еженедельная резервная копия Sheets вручную (`Файл → Создать копию`). История версий Google хранит 30 дней |
| Опечатка в имени подрядчика в колонке `*СП` | Data Validation на `Подрядчики!A:A` (§7.9) блокирует ввод свободным текстом |
| Опечатка в заголовке колонки работы в `Главный` (рассинхрон с `Работы.Полное название`) | При старте GAS читает обе таблицы и логирует расхождения. UI показывает баннер «Структура листа Главный не соответствует справочнику работ, обратитесь к администратору» |
| Падение GAS Web App (квота / ошибка) | Подрядчик видит попап «Не удалось загрузить, попробуйте позже». Админ работает напрямую в Sheets — UI не блокирует основные процессы |
| Конкурентная запись (два сотрудника одной команды) | `LockService` в GAS (§6) |
| Подрядчик не получает обновления при изменении назначений админом | Кнопка «Обновить» в UI. Кеш сбрасывается автоматически если структура колонок поменялась (`version_hash`, §6) |
| Сбой системного времени на ПК подрядчика | Дата ставится сервером GAS, не клиентом (§7.16) |
