# ТЗ: Веб-приложение учёта работ по паркингу

## 1. Назначение

Веб-приложение для подрядчиков и админа — отметка факта выполнения работ по ~2000 помещений **одного** паркинга. Каждый пользователь заходит по своей ссылке-токену; интерфейс — единый, права различаются.

**Целевая платформа — десктоп** (мобильную адаптацию не делаем). **Один токен = вся команда подрядчика** (не персональный, идентифицируем подрядчика как юр.лицо).

**Три роли в UI:**
- **Подрядчик** — видит только помещения, где он назначен хотя бы на одну работу. Отмечает выполнение только своих назначений. Чужие назначения видны read-only с именем подрядчика (в попапе).
- **Админ** (`ИД_подрядчик = admin`) — видит все помещения и все назначения, может ставить/снимать отметки за любого подрядчика. Управление реестром, справочниками и назначениями (колонки `*СП`) — только в Google Sheets.
- **Наблюдатель** (`ИД_подрядчик = viewer`) — read-only. Видит всё как админ, но без возможности что-либо менять. Бэк отклоняет любые write-запросы.

## 3. Структура Google Sheets

### Лист `Главный` — реестр + назначения + факт в одной таблице

Одна строка = одно помещение. Слева реестровые поля, справа парные колонки на каждую работу.

**Реестровые колонки (A–L)** — заполняет/копирует админ при импорте помещений:

| Колонка | Поле | Источник |
|---|---|---|
| A | Корпус | из исходного реестра |
| B | Этаж | из исходного реестра |
| C | Номер (PK, `7.-3.01`) | из исходного реестра |
| D | Наименование | из исходного реестра |
| E | Площадь, м.кв. | из исходного реестра |
| F | Периметр, м.п. | из исходного реестра |
| G | Площадь стен | из исходного реестра |
| H | Категория помещения | из исходного реестра |
| I | Наименование чертежа | из исходного реестра |
| J | Захватка | из исходного реестра |
| K | Комментарий (админский, подрядчик не видит) | вручную |
| L | Распределение для КС (группа для применимости работ) | из исходного реестра |

**Парные колонки на каждую работу (M+), повторяются для всех работ из листа `Ведомость_работ` в порядке строк:**

- `<Название работы>` — *дата факта*. Хранится в Sheets как **`Date` (формат ячейки = `Дата`, отображение `ДД.ММ.ГГГГ`)**. Пусто = не сделано.
- `<Название работы> СП` — *подрядчик* (имя из листа `Ведомость_подрядчиков`). Пусто = работа не нужна в этом помещении.

Пример для работы «Гидроизоляция полов»:

| ... | Гидроизоляция полов | Гидроизоляция полов СП | Заливка стяжки | Заливка стяжки СП | ... |
|---|---|---|---|---|---|
| ... | 20.05.2026 | Спектр | | Спектр | ... |
| ... | | | 20.05.2026 | Спектр | ... |

**Правила колонок:**
- A–L редактирует только админ в Sheets
- Колонки `*СП` (назначение) — ведёт админ в Sheets
- Колонки дат — заполняются подрядчиком через UI (либо админом в Sheets вручную)
- **Применимость работы** к помещению = «есть значение в колонке СП». Нет СП = работа не нужна тут.
- **Инвариант**: дата не может быть заполнена при пустом СП. Условное форматирование подсвечивает нарушение красным.

**Правила работы GAS с листом `Главный` (критично, реестр — боевые данные):**
- **GAS никогда не пишет в столбцы A–L** (реестровая часть). Только M+ (колонки работ и СП).
- **GAS не добавляет и не удаляет строки** в листе. Только обновляет существующие ячейки в M+.
- В коде это enforce'ится явной проверкой: `if (colIndex <= 12) throw new Error('Запись в реестр запрещена')`. Любая ошибка вычисления индекса = падение с понятной ошибкой, а не порча данных.
- Лист с реестром в файле называется ровно **`Главный`** (точное имя для `getSheetByName`).
- На момент старта разработки последняя занятая колонка работ — `Заливка топпинга СП` (после неё новых работ нет, можно добавлять справа без конфликтов).

### Лист `Ведомость_работ` — справочник, ведётся вручную

| A: Полное название | B: Поверхность | C: Название для площадки | D: ИД_работы |
|---|---|---|---|
| Гидроизоляция полов | Полы | Гидроизоляция | down_hydro |
| Заливка стяжки | Полы | Стяжка | down_st |
| Укладка плитки | Полы | Плитка | down_kerama |
| Покраска потолка | Потолки | Покраска потолок | up_color |
| Штукатурка стен | Стены | Штукатурка | wall_stukat |
| Покраска стен | Стены | Покраска стены | wall_paint |
| Монтаж двери | Двери | Двери | doors |
| Заливка топпинга | Полы | Топпинг | down_topping |

- **A: Полное название** — официальное имя работы. **Должно точно совпадать с заголовком колонки в `Главный`** (например `Гидроизоляция полов` / `Гидроизоляция полов СП`). При переименовании работы — переименовать одновременно тут и в `Главный`.
- **B: Поверхность** — группа работ (`Полы` / `Стены` / `Потолки` / `Двери`). Используется для группировки колонок в UI подрядчика с подзаголовками.
- **C: Название для площадки** — короткое имя для заголовков колонок UI подрядчика (узкие столбцы на мобилке). Можно править без последствий для системы.
- **D: ИД_работы** — стабильный латинский идентификатор для API между фронтом и GAS. **Неизменяемый.** Используется в URL/JSON запросах.
- Порядок строк в листе = порядок колонок в `Главный` и в UI (нет отдельной колонки «порядок»).

### Лист `Ведомость_подрядчиков` — справочник, ведётся вручную

| A: Подрядчик | B: ИД_подрядчик | C: Статус | D: Токен |
|---|---|---|---|
| Овчинников | ovchi | Активен | 81bc3299c7ceb6fb |
| Спектр | spectr | Активен | ce3d0facb5187091 |
| Альфа Строй | alfa | Активен | 8ca318c5521b5b66 |
| Админ | admin | Активен | <admin_token_16hex> |
| Наблюдатель | viewer | Активен | <viewer_token_16hex> |

- **A: Подрядчик** — имя как пишется в колонках `*СП` листа `Главный`. **Primary key** для связи с реестром. Может содержать пробелы и кириллицу.
- **B: ИД_подрядчик** — стабильный латинский ID для API/кода. Используется в логах и при сравнении на бэке. Регистр единообразный (строчные). **Зарезервированное значение: `admin` = роль администратора** (бэк даёт права на отметку любых работ за любого подрядчика).
- **C: Статус** — текстовое значение. **`Активен` = доступ открыт; любое другое значение (или пусто) = доступ закрыт** (можно использовать `Уволен`, `На паузе` и т.п. — бэк сравнивает строго на равенство `"Активен"`).
- **D: Токен** — 16 hex-символов, генерируется формулой (см. §7). Уникальный ключ доступа подрядчика, передаётся в URL.
- Колонка «какие работы делает» **не нужна** — применимость определяется фактическими записями в `Главный.*СП`.
- Запись с `ИД_подрядчик = admin` — не должна попадать в колонки `*СП` листа `Главный` (админ ничего не делает, он только проверяет/корректирует).
- Запись с `ИД_подрядчик = viewer` — роль **Наблюдатель**: видит всё как админ, но не может ставить/снимать отметки. Бэк отклоняет любые write-запросы от viewer'а с ошибкой «У роли Наблюдатель нет прав на редактирование».

### Лист `Типы_помещений` — справочник типов с цветами и группировкой

Используется для (1) цветовой индикации карточек в матрице (см. §5), (2) группирующего фильтра «Тип помещения» в шапке UI.

| A: Тип_помещения_детально | B: Тип_помещения_панель | C: Цвет |
|---|---|---|
| Кладовая | Кладовая | Желтый |
| Тамбур-шлюз (ЛХ/ПБЗ) | ТШ | (пусто) |
| Лестничная клетка | ЛК | (пусто) |
| Архив | Прочие помещения | (пусто) |
| Душевая охраны | Прочие помещения | (пусто) |
| Помещение КНС | Тех помещения | Серый |
| Автостоянка | Автостоянка | (пусто) |
| Лифтовой холл | ЛХ | (пусто) |
| Тамбур | Тамбур | (пусто) |

- **A: Тип_помещения_детально** — точное побайтное совпадение со значениями в `Главный!D` (`Наименование`). Каждое уникальное наименование из реестра должно иметь строку здесь.
- **B: Тип_помещения_панель** — укрупнённая категория для фильтра в UI. Несколько детальных типов могут мапиться в одну панельную категорию (например, все «Прочие помещения» агрегируются под одной чипсой фильтра).
- **C: Цвет** — текстовое название цвета. Фронт мапит на hex через статический пресет в коде. Пустое значение или название вне пресета — дефолтный светло-серый (`#E5E7EB`).
  - **Зелёный цвет (любой оттенок) НЕЛЬЗЯ использовать** для типа помещения — он зарезервирован за индикацией «работа готова» (см. §5). Если в Sheets будет введено `Зелёный` — фронт игнорирует и применяет дефолтный серый.
  - Пресет цветов в коде (расширяемый): Жёлтый, Серый, Оранжевый, Красный, Синий, Розовый, Фиолетовый, Голубой, Коричневый. Конкретные hex подбираются при разработке так чтобы белый текст номера читался поверх.
- **Растущий лист** — появилось новое уникальное `Наименование` в реестре → админ добавляет строку. Если строки нет — фронт показывает карточку с дефолтным серым и логирует warning в консоли GAS.

**Связь типов с реестром:** GAS при формировании JSON делает lookup на лету: для каждой строки `Главный` по полю `Наименование` ищет соответствующую строку в `Типы_помещений`, добавляет в JSON помещения два поля: `тип_панель` и `цвет` (как текст, фронт мапит на hex). Колонок в `Главный` не добавляем — `Главный` остаётся чистым.

## 4. Авторизация и роли

Без паролей. Первичный доступ по ссылке: `https://nick3000ept.github.io/parking-admin/?t=<token>`

- **Подрядчик**: токен → запись в `Ведомость_подрядчиков` (если `Статус == "Активен"`) → его `Подрядчик` (имя). Видит **только помещения, где он назначен хотя бы на одну работу**. Активно работать может только со своими назначениями. Чужие назначения видны read-only с именем подрядчика в попапе.
- **Админ**: токен с `ИД_подрядчик = admin` (если `Статус == "Активен"`). Видит все помещения, **может ставить/снимать отметки за любого подрядчика** в любой работе где есть назначение. Назначать СП — по-прежнему только в Sheets.
- **Наблюдатель**: токен с `ИД_подрядчик = viewer` (если `Статус == "Активен"`). Видит всё как админ, но без прав на запись — любой write-запрос отклоняется бэком.
- Без токена / невалидный / `Статус != "Активен"` → экран «Доступ закрыт».

**Жизненный цикл токена в браузере:**
- При первом заходе по `?t=<token>` фронт сохраняет токен в `localStorage` под ключом `parking_token` и **удаляет `?t=` из URL** (через `history.replaceState`). Защита от утечки через скриншоты, history и расшаренные ссылки.
- При последующих заходах токен берётся из `localStorage`. URL остаётся чистым.
- Кнопка «Выйти» удалена из UI (текущая итерация). Принудительный logout — `localStorage.clear()` в DevTools или приватная вкладка.

**Ротация токена при утечке** (см. §7.7 и §7.14):
- Админ в Sheets перегенерирует значение в `Подрядчики.Токен` через формулу
- Старый токен мгновенно перестаёт работать (бэк не находит совпадения)
- Новую ссылку отправляют легитимному подрядчику
- При следующем заходе со старым токеном — экран «Доступ закрыт»

**Безопасность** — проверка на бэке. На каждый запрос GAS сверяет:
1. Токен валиден и `Статус == "Активен"`
2. **Если роль `viewer`** — любой write-запрос отклоняется до проверки прав.
3. Для write-запроса (Номер, ИД_работы):
   - Если **подрядчик**: в листе `Главный` строка с этим Номером, колонка `<название_работы> СП` содержит **ровно его имя** (после `.trim()`). Иначе — отказ.
   - Если **админ** (`ИД_подрядчик == 'admin'`): достаточно чтобы колонка `<название_работы> СП` была непустой (есть кому назначено). Иначе — отказ.
4. **Менять СП запрещено всем** (включая админа в UI) — только через Sheets. UI работает только с колонкой даты.
5. **Редактировать дату нельзя** — только полный clear+set (бинарно «есть/нет»).
6. **Имена сравниваются после `.trim()`** на бэке — защита от случайных trailing-пробелов в Sheets.

#### Шапка (sticky, тёмно-серая `#1F2937`)

- Название проекта `Паркинг СБ3`
- **Бейдж пользователя** сразу после названия:
  - Подрядчик — синий
  - Админ — красный, с пометкой `· АДМИН`
  - Наблюдатель — серый, с пометкой `· ПРОСМОТР`
- **Прогресс — двумя строками с цветовой разбивкой**:
  - `ПОМЕЩЕНИЙ  N · M(зелёное «готово») · K(жёлтое «остаток»)  «работа»`
  - `ПЛОЩАДЬ    Σ · Σ_done(зелёное) · Σ_rest(жёлтое)  м²`
  - Считается по отфильтрованным помещениям. Если работа выбрана — «готово/остаток» по этой работе; иначе — по всем работам пользователя (или всем работам всех подрядчиков для админа/наблюдателя)
  - Числа форматируются с разделителями тысяч ru-RU (`10 000`)
- **Кнопка «× Очистить фильтры»** — серая, сбрасывает все фильтры в дефолт
- **Кнопка «Режим отметки»** — ярко-зелёная всегда; при активном режиме становится тёмно-зелёной с тенью и рамкой «вдавленной» кнопки. В режиме отметки фон шапки становится `#047857`
- **Кнопок «Обновить» и «Выйти» нет.** Свежие данные подгружаются при загрузке страницы. Logout — через `localStorage.clear()` в DevTools / приватная вкладка

#### Фильтры (тёмная панель `#1F2937`, лейблы inline слева светло-серые)

- **Захватка** — чипсы из уникальных значений `Главный!J`. Multi-select.
- **Корпус** — чипсы из `Главный!A`, сортировка: сначала числа (7…12), потом буквенные (П1, П2…). Multi-select.
- **Этаж** — чипсы из `Главный!B`, по возрастанию (-3, -2, -1). Multi-select.
- **Тип помещения** — dropdown (single-select) из уникальных значений `Тип_помещения_панель` листа `Типы_помещений`. Опция «— Все типы —» по умолчанию.
- **Поиск по номеру и названию** — два узких input на одной строке. Левый ищет по `Главный!C` (частичное совпадение), правый — по `Главный!D` или `Главный!L`.

**На тёмном фоне:** чипсы полупрозрачные (`rgba(255,255,255,0.08)`), активная — белая с тёмным текстом; input и select — прозрачно-тёмные с белым placeholder.

#### Чипсы работ (белая полоса между фильтрами и матрицей)

- `[Все работы]` — слева, single-select toggle (повторный клик по активной — возврат к «Все»).
- Далее работы из листа `Ведомость_работ`, **сгруппированные по `Поверхность`** (порядок групп — по первому появлению в листе): Полы, Стены, Потолки, Двери. Группы разделены тонкой вертикальной чертой `│`. Текстовых подзаголовков (`ПОЛЫ` / `СТЕНЫ`) нет — поверхность видна в tooltip при наведении.
- Используется `Название для площадки` (короткое имя).
- Для **подрядчика** — показываются только работы, на которые он назначен хотя бы где-то. Для **админа/наблюдателя** — все работы.
- **Тогл «Скрыть готовые»** прижат к правому краю той же полосы. Активен только когда работа выбрана: скрывает помещения с заполненной датой по выбранной работе.

#### Матрица этаж × корпус

- **Layout**: CSS Grid. Первая колонка — узкие лейблы этажей (40px); далее по колонке на каждый корпус (`1fr`).
- **Сетка строк**: `grid-auto-rows: minmax(120px, max-content)` — высота строки = по содержимому ячейки, минимум 120px. **Нет растяжения «на весь экран»** — много карточек = высокая матрица, мало — короткая. Страница скроллится естественно.
- **Сортировка корпусов**: числовые по возрастанию, затем буквенные (`7,8,…,12,П1,П2`). **Этажи**: по возрастанию (`-3` сверху, `-1` внизу — поправлено: фактически визуальный порядок снизу-вверх, нумерация номера помещения сохраняется).
- **Заголовки**: над колонками номера корпусов (`К7`, `К8`, …). Лейблы этажей — слева в каждой строке (`-3`, `-2`, `-1`).
- **Ячейка матрицы** — белая, с тонкой границей. Внутри `flex wrap` для карточек, направление left-to-right, top-to-bottom. Скролла внутри ячейки нет — высота растёт под контент.
- **Карточка помещения** — фиксированный размер 96×52px:
  - Строка 1 — номер помещения (`7.-3.01`), 14px bold
  - Строка 2 — тип помещения (`Тип_помещения_панель` из листа `Типы_помещений`), 9px, opacity 0.75, ellipsis при переполнении
  - Фон карточки см. ниже

#### Состояния карточек

**Работа НЕ выбрана:**

## 6. Производительность и параллелизм

### Объём данных
- ~2000 помещений × ~10 работ × парные колонки (дата + СП) = до 40k значений по работам + ~12 колонок реестра
- Сырой JSON для матрицы — оценочно ~1–1.5 MB. С gzip от GAS — ~300–500 KB по сети. Терпимо.

### Бэк (GAS)
- **doGet**: читает `Главный` целиком (один `getValues`), листы `Ведомость_работ`, `Ведомость_подрядчиков`, `Типы_помещений`. Делает lookup типа помещения по `Наименование` → `Тип_помещения_панель` и `Цвет`. Сериализует в JSON. На фронт отдаёт **всё** — права проверяются на write, не на read. Ожидаемое время — 3–8 сек.
- **doPost** — fast-path с подсказкой:
  - Фронт передаёт `hint = {row, colDate, colSp}` из снапшота. Бэк делает один батч-read нужной строки, проверяет что `Главный!C[row]` совпадает с ожидаемым `num` (защита от устаревшего hint), читает SP и текущую дату.
  - Если hint протух или не передан — fallback на полный `locateCell` (линейный поиск).
  - Цель — <500мс на запрос.
- **LockService** — берётся **только на сам `setValue`/`clearContent`** (~50мс), не на весь doPost. Read-операции (auth, чтение SP/date) идут параллельно без блокировки. `waitLock(30000)` — таймаут 30 сек на случай очереди.
- **Идемпотентность** (см. §7.5): повторная установка даты на уже отмеченную ячейку — no-op. Повторное снятие на уже пустую — no-op.
- Без поллинга/автообновления — обновление при F5 / `loadData(true)`.

### Фронт
- **Optimistic UI**: при клике в режиме отметки карточка мгновенно меняет визуал (без ожидания сервера). Параллельно идёт fetch. Если сервер вернул ошибку — откат + toast.
- **Targeted refresh**: при изменении одной ячейки перерисовывается **только эта одна карточка** (`refreshCard(num)`), а не вся матрица из 2000 карточек.
- **Sequence на ячейку** (`cellSeq[num+'|'+idRaboty]`) — защита от race condition при быстром set→clear на одной ячейке: ответ от сервера применяется только если `mySeq` совпадает с текущим. Устаревшие ответы игнорируются.
- **2000 карточек в матрице** — CSS Grid с фиксированным размером карточек справляется без виртуализации на десктопе.
- **Фильтрация** — рендерится только видимая часть после `matchesFilters`; неактуальные карточки в DOM не попадают.
- **Saveкэш debounced 300мс** — JSON.stringify ~1MB снапшота не блокирует клик.

### Кеш в `localStorage`
- Ключ: `parking_data_<token>`
- Содержимое: полный JSON ответа `doGet` + `version_hash` (хеш от заголовков `Главный` + содержимого справочников)
- Стратегия: при открытии страницы UI рендерит из кеша **мгновенно**, параллельно делает `doGet`. После ответа `state.data` перезаписывается из свежих данных, render повторяется.

## 7. Правила разработки

1. **Источник правды о работах** — лист `Ведомость_работ`, колонка `ИД_работы` = стабильный латинский ID для API.
2. **Заголовки колонок работ в `Главный`** должны точно совпадать с `Ведомость_работ.Полное название` (например `Гидроизоляция полов` / `Гидроизоляция полов СП`). При переименовании работы — обновить оба места одновременно. Колонки `Поверхность` и `Название для площадки` можно править без последствий.
3. **Источник правды о подрядчиках** — лист `Ведомость_подрядчиков`, колонка `Подрядчик` = primary key. Имена в `Главный.*СП` должны совпадать побайтно (бэк делает `.trim()` при сравнении).
4. **Валидация прав на бэке**: см. §4. Клиенту не доверяем.
5. **Идемпотентность**: повторная отправка той же даты на ту же ячейку не дублирует.
6. **Оффлайн**: при обрыве сети — попап «не удалось сохранить, повторить?». Локально не пишем.
7. **Генерация токенов**: формула в листе `Ведомость_подрядчиков`, колонка `Токен` (16 hex-символов):
   ```
   =LOWER(DEC2HEX(RANDBETWEEN(1,4294967295),8))&LOWER(DEC2HEX(RANDBETWEEN(1,4294967295),8))
   ```
   После генерации — Ctrl+Shift+V → значения, чтобы не пересчиталось.
8. **Деплой фронта**: `git add . && git commit && git push` → GitHub Pages обновится через ~1 мин. **Деплой GAS**: через clasp — `clasp push --force && clasp update-deployment --description "vX" <DEPLOYMENT_ID>`. URL остаётся прежним.
9. **Data Validation в `Главный`**: колонки `*СП` — выпадающий список из `Ведомость_подрядчиков.Подрядчик`. Защита от опечаток.
10. **Условное форматирование в `Главный`**:
    - Серый фон колонки `<работа>` когда `<работа> СП` пуста — визуально «работа не нужна тут»
    - Красный фон ячейки даты если СП пуст (нарушение инварианта)
11. **При импорте новых помещений** из исходного реестра — копируются только колонки A–J и L. Колонки K (комментарий) и работ (M+) остаются пустыми, заполняются в `Главный`.
12. **Добавление новой работы в систему** (рутинная операция, список работ растущий):
    1. Добавить строку в листе `Ведомость_работ`: `Полное название`, `Поверхность`, `Название для площадки`, латинский `ИД_работы`
    2. В листе `Главный` добавить **две новые колонки в конец**: `<Полное название>` и `<Полное название> СП`. Порядок колонок в `Главный` должен соответствовать порядку строк в `Ведомость_работ`.
    3. Назначить подрядчиков в колонке `*СП` где работа применима
    4. UI подхватит новую работу автоматически при следующем F5
    
    Удалять колонки уже использованных работ нельзя (потеряются исторические даты). Работу можно «деактивировать» удалением строки в `Ведомость_работ` — в `Главный` колонки останутся, UI их игнорирует.
13. **Добавление нового подрядчика** (рутинная операция, список растущий):
    1. Добавить строку в листе `Ведомость_подрядчиков`: `Подрядчик` (имя), `ИД_подрядчик` (латиница строчными — `admin` или `viewer` зарезервированы для ролей), `Статус = Активен`, `Токен` (через формулу, потом Ctrl+Shift+V)
    2. Имя автоматически появится в выпадающих списках колонок `*СП` листа `Главный` (Data Validation на диапазоне `Ведомость_подрядчиков!A:A`)
    3. Передать подрядчику ссылку с его токеном
    
    «Уволить» подрядчика — поставить `Статус = Уволен` (или любое значение, отличное от `Активен`). Назначения в `Главный` остаются (исторические данные).
14. **Ротация токена** (при подозрении на утечку):
    1. В листе `Ведомость_подрядчиков`, ячейка `Токен` нужного подрядчика — вставить формулу из §7.7 заново
    2. Ctrl+Shift+V → значения (превратить в строку)
    3. Скопировать новый токен, отправить подрядчику новую ссылку
    4. Старая ссылка перестаёт работать мгновенно — старый кеш в браузере у злоумышленника даст одну попытку чтения, после чего бэк отклонит токен. **Локальные правки злоумышленник сделать не сможет** (бэк всегда сверяет токен на каждый запрос).
15. **Смена СП в Главный при существующей дате**: дата НЕ очищается. Это считается передачей записи новому подрядчику. UI старого подрядчика перестаёт видеть эту ячейку (она больше не его), новый подрядчик видит её уже отмеченной и может снять/перепоставить.
16. **Дата ставится сервером**: при `doPost` действия `setDate` GAS подставляет `new Date()` в таймзоне `Europe/Moscow`. Клиент не передаёт дату.
17. **Удаление помещения из реестра**: если админ удаляет строку в листе `Главный`, все проставленные даты и назначения по этому помещению теряются вместе со строкой. Бэкап — ответственность админа (см. §11).

## 8. Чего НЕ делаем (выкинуто намеренно)

- Промежуточные статусы — только «есть дата»/«нет даты». **Подтверждение/приёмка админом не предусмотрены** (дата подрядчика = факт)
- **Редактирование даты подрядчиком** — только бинарно поставить/снять, всегда сегодняшняя
- Фото-доказательства
- Лог действий (кто когда что нажал) — не ведём. При спорах источник — Sheets, версии Google можно посмотреть через `Файл → История версий`
- Уведомления (push, email, Telegram)
- **Мобильная и планшетная версии** — только десктоп (min-width 1024px)
- PWA / offline-режим
- Регистрация / сброс доступа подрядчиками (только админ ротирует токен)
- Управление справочниками из UI (только в Sheets)
- **Назначение подрядчиков (колонки `*СП`) через UI** — даже у админа. Только в Sheets.
- Отдельная матрица применимости работ (применимость = есть подрядчик в `*СП`)
- Отдельные нормализованные листы `Назначения` / `Статусы` (всё в `Главный`)
- Комментарии подрядчика к отдельной отметке (общий комментарий ведёт админ в колонке K)
- Колонка «какие работы делает подрядчик» в листе `Подрядчики`
- **Мультитенантность** — система только под один паркинг
- **Персональные токены сотрудников** — токен на команду подрядчика, не на физлицо

## 10. Риски и операционные меры

| Риск | Меры |
|---|---|
| Утечка токена подрядчика (скриншот/расшаренная ссылка) | Очистка `?t=` из URL после первого входа (§4). Ротация через перегенерацию формулы (§7.14). Влияние ограничено — злоумышленник может менять даты только на ячейках одного подрядчика |
| **Утечка админского токена** — высокий риск, злоумышленник может менять любые даты | Админ-токен хранится только у заказчика, не передаётся через мессенджеры/email. При подозрении на утечку — немедленная ротация. Не использовать admin-токен на чужих устройствах |
| Случайное удаление строки в `Главный` | Еженедельная резервная копия Sheets вручную (`Файл → Создать копию`). История версий Google хранит 30 дней |
| Опечатка в имени подрядчика в колонке `*СП` | Data Validation на `Ведомость_подрядчиков!A:A` (§7.9) блокирует ввод свободным текстом. На бэке имена сравниваются после `.trim()` (защита от trailing-пробелов). |
| Опечатка в заголовке колонки работы в `Главный` (рассинхрон с `Ведомость_работ.Полное название`) | При doGet GAS строит `workColMap` по точному совпадению заголовка с `Полное название`. Работа без колонок просто не появится в UI. |
| Падение GAS Web App (квота / ошибка) | Подрядчик видит попап «Не удалось загрузить, попробуйте позже». Админ работает напрямую в Sheets — UI не блокирует основные процессы |
| Конкурентная запись (два сотрудника одной команды) | `LockService` в GAS (§6) |
| Подрядчик не получает обновления при изменении назначений админом | F5 на странице — фронт делает форсированный fetch и обновляет state. Кеш сбрасывается автоматически если `version_hash` изменился (§6) |
| Сбой системного времени на ПК подрядчика | Дата ставится сервером GAS, не клиентом (§7.16) |
